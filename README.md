# Лабораторная работа № 4. Транслятор и модель процессора

**Вариант:**

```
alg | risc | neum | hw | tick | binary | trap | port | pstr | prob1 | -
```

| Тег | Значение в работе |
|-----|-------------------|
| `alg` | Язык в стиле Java/JavaScript: переменные, выражения, `if`/`while`, функции. Транслятор строит человекочитаемый AST, проверяемый в тестах. |
| `risc` | Упрощённый набор команд: фиксированная длина 4 байта, АЛУ работает только с регистрами, доступ к памяти и портам — отдельными командами. |
| `neum` | Архитектура фон Неймана: единая память для кода и данных. |
| `hw` | Hardwired control unit — устройство управления реализовано кодом модели. |
| `tick` | Потактовое моделирование: каждый такт — отдельная итерация, журнал ведётся по тактам. |
| `binary` | Настоящий бинарный образ `.bin` + отладочный листинг `.lst`. |
| `trap` | Ввод-вывод через систему прерываний; обработчик пишется на самом языке. |
| `port` | Port-mapped I/O: отдельные команды `IN`/`OUT` с номером порта. |
| `pstr` | Строки с префиксом длины (Pascal string), один символ — одно машинное слово. |
| `prob1` | Project Euler #4 (наибольший палиндром-произведение). |

## 1. Язык

### 1.1. Особенности

- C-подобный синтаксис, выражения с приоритетами операций.
- Скалярные переменные (`var`) и массивы фиксированного размера (`var a[N]`).
- Управляющие конструкции `while`, `if`/`else`.
- Функции (`function`) с параметрами и возвратом значения; функции-обработчики прерываний (`interrupt function`) без параметров и без возврата значения.
- Встроенные процедуры ввода-вывода и работы с прерываниями.
- Комментарии `// до конца строки`.
- Строковые литералы хранят символы как есть (escape-последовательности не интерпретируются; перевод строки выводится как `print(10)`).
- Унарного минуса и логических `&&`/`||` нет; сравнения возвращают `0`/`1`.

### 1.2. Грамматика (BNF)

```bnf
<program>     ::= { <function> | <statement> }
<function>    ::= [ "interrupt" ] "function" <ident> "(" [ <params> ] ")" <block>
<params>      ::= <ident> { "," <ident> }
<block>       ::= "{" { <statement> } "}"

<statement>   ::= <var-decl> | <array-decl> | <assign> | <array-assign>
                | <while> | <if> | <print> | <print-num> | <print-str>
                | <return> | <call>
<var-decl>    ::= "var" <ident> "=" <expr>
<array-decl>  ::= "var" <ident> "[" <number> "]"
<assign>      ::= <ident> "=" <expr>
<array-assign>::= <ident> "[" <expr> "]" "=" <expr>
<while>       ::= "while" "(" <expr> ")" <block>
<if>          ::= "if" "(" <expr> ")" <block> [ "else" <block> ]
<print>       ::= "print" "(" <expr> ")"          // вывести младший байт как символ
<print-num>   ::= "print_num" "(" <expr> ")"      // вывести число в десятичном виде
<print-str>   ::= "print_str" "(" <string> ")"    // вывести статическую строку
<return>      ::= "return" [ <expr> ]
<call>        ::= <ident> "(" [ <args> ] ")"

<expr>        ::= <add> { ("==" | "!=" | "<" | "<=" | ">" | ">=") <add> }
<add>         ::= <term> { ("+" | "-") <term> }
<term>        ::= <factor> { ("*" | "/" | "%") <factor> }
<factor>      ::= <number> | "(" <expr> ")" | <ident>
                | <ident> "(" [ <args> ] ")"      // вызов функции / builtin
                | <ident> "[" <expr> "]"          // чтение элемента массива
```

### 1.3. Встроенные процедуры

| Вызов | Назначение |
|-------|-----------|
| `port_in(N)` | прочитать байт из порта `N` (`N` — константа) |
| `port_out(N, v)` | записать `v` в порт `N` |
| `set_interrupt_handler(vec, fn)` | назначить функцию `fn` обработчиком вектора `vec` |
| `enable_interrupts()` / `disable_interrupts()` | разрешить / запретить прерывания |
| `memread(addr)` / `memwrite(addr, v)` | прямой доступ к памяти по адресу |

### 1.4. AST

AST — обычные словари/списки Python, отображаемые в человекочитаемый вид функцией `format_ast` (артефакт `.ast`). Пример для `var x = 2 + 3 * 4`:

```
Program
  body:
    VarDecl x =
      BinOp +
        Num 2
        BinOp *
          Num 3
          Num 4
```

Видно, что приоритет операций (`*` глубже `+`) отражён структурой дерева. Корректность AST проверяется тестами в `tests/test_ast.py`.

## 2. Организация памяти

Память единая (фон Нейман), байтовая адресация, машинное слово — 4 байта, однопортовая (за такт выполняется не более одного обращения). Размер — 32 КБ.

| Диапазон | Назначение |
|----------|-----------|
| `0x0000 … 0x2000` | секция кода (до 2048 инструкций) |
| `0x2000 … 0x22F0` | глобальные/локальные переменные и массивы (bump-аллокатор) |
| `0x22F0 … 0x2300` | таблица векторов прерываний (4 вектора × 4 байта) |
| `0x2300 … 0x2400` | служебный буфер цифр для `print_num` |
| `0x2400 … 0x4000` | статические строки (pstr) |
| `0x4000 … 0x8000` | стек (растёт вверх, `R7` = SP) |

Статические строки хранятся в формате pstr: первое слово — длина, далее по одному символу на слово.

## 3. Система команд (ISA)

### 3.1. Формат инструкции

Фиксированные 32 бита:

```
 31    24 23  20 19  16 15            0
+--------+------+------+----------------+
| opcode |  rd  |  rs  |   imm (16)     |
+--------+------+------+----------------+
```

- `opcode` — код операции (8 бит);
- `rd`, `rs` — регистры приёмника и источника (по 4 бита, регистров 8);
- `imm` — знаковое 16-битное непосредственное значение / адрес перехода / номер порта.

Инструкция кодируется big-endian; слова данных в памяти — little-endian.

### 3.2. Набор команд

| Код | Мнемоника | Семантика |
|-----|-----------|-----------|
| 1 | `add rd, rs` | `rd ← rd + rs` |
| 2 | `sub rd, rs` | `rd ← rd - rs` |
| 3 | `mul rd, rs` | `rd ← rd * rs` |
| 4 | `div rd, rs` | `rd ← rd / rs` |
| 5 | `mod rd, rs` | `rd ← rd % rs` |
| 6 | `cmp rd, rs` | выставить флаги по `rd - rs` |
| 7 | `ld rd, [rs+imm]` | `rd ← mem[rs+imm]` |
| 8 | `st rd, [rs+imm]` | `mem[rs+imm] ← rd` |
| 9 | `li rd, #imm` | `rd ← imm` |
| 10 | `jmp addr` | безусловный переход |
| 11–16 | `jz/jnz/jl/jle/jg/jge addr` | переходы по флагам |
| 17 | `in rd, port#imm` | `rd ← port[imm]` |
| 18 | `out rs, port#imm` | `port[imm] ← rs` |
| 19 | `halt` | останов |
| 20 | `iret` | возврат из прерывания |
| 21 | `ei` / 22 `di` | разрешить / запретить прерывания |
| 23 | `jr rs` | переход по адресу из регистра |

АЛУ-операции работают только с регистрами; единственный доступ к памяти — `ld`/`st`; к портам — `in`/`out`. Это соответствует требованию RISC.

Все АЛУ-результаты приводятся к знаковому 32-битному диапазону (`_mask32`), имитируя переполнение реального 32-битного процессора.

## 4. Транслятор

### 4.1. Запуск

```
python translator.py <исходник.alg> <выход.bin>
```

Создаёт три артефакта: `<выход>.bin` (бинарный образ), `<выход>.lst` (листинг), `<выход>.ast` (дерево).

### 4.2. Фазы

1. **Лексер** (`tokenize`) — поток токенов.
2. **Парсер** (`parse`) — AST.
3. **Кодген** (`CodeGen.generate`) — список инструкций + секция данных.
4. **Линковка** (`link`) — разрешение меток в адреса.
5. **Сборка образа** — инструкции с адреса 0, данные по своим адресам; запись `.bin`, `.lst`, `.ast`.

### 4.3. Отображение сложных выражений на регистры и память

Выражения вычисляются по **стековой модели**, что упрощает кодген произвольной вложенности:

- операнды кладутся на программный стек (память, `R7` = SP);
- бинарная операция снимает два верхних значения в `R1`/`R2`, выполняет АЛУ-операцию, кладёт результат обратно;
- `R3` используется как временный регистр в `push`/`pop`.

Переменные размещаются в секции данных; чтение — `ld`, запись — `st`. Элемент массива `a[i]` транслируется в адрес `base + i*4`, вычисляемый в регистре, после чего идёт `ld`/`st` со смещением 0.

Пример: `x = 2 + 3 * 4` сначала вычисляет `3 * 4` (вложенный `term`), затем `2 + (…)`, и записывает результат в ячейку переменной `x`. Порядок вычисления полностью задаётся структурой AST.

## 5. Модель процессора

### 5.1. Запуск

```
python machine.py <образ.bin> [файл_ввода] [--log <файл>] [--quiet]
```

Файл ввода — расписание прерываний, строки `<такт> <символ>` (символ — буква, число-код или `\n`, `\0`).

### 5.2. Ресурсы

- 8 регистров общего назначения `R0…R7` (`R7` — указатель стека);
- флаги `ZF` (ноль) и `NF` (знак);
- единая память 32 КБ.

### 5.3. Потактовая модель

Каждая инструкция проходит две фазы, по такту на каждую: `FETCH` (выборка и декодирование) и `EXEC` (исполнение). Вход в прерывание занимает отдельный такт `TRAP`. Моделирование можно прервать на любом такте.

Устройство управления **hardwired** — логика декодирования и исполнения зашита в модели (`run`).

### 5.4. Прерывания (trap)

- Запрос прерывания проверяется **только на границе инструкций** (в фазе `FETCH`).
- Адрес обработчика берётся из таблицы векторов; обработчик задаётся на языке через `set_interrupt_handler` и заканчивается `iret`.
- При входе сохраняется полный контекст: `PC`, **все регистры и флаги**; при `iret` он восстанавливается.
- Вложенные прерывания **маскируются**: на входе прерывания запрещаются, повторный запрос откладывается и срабатывает после `iret`.
- В журнале состояние процессора (`RUN`/`INT`/`STOP`) показывается в каждой строке, поэтому всегда видно, идёт ли работа в прерывании.

### 5.5. Ввод-вывод (port-mapped)

- Порт 0 — ввод (клавиатура): данные приходят по расписанию прерываний.
- Порт 1 — вывод: запись добавляет символ в выходной буфер.
- «Получение данных» и «вызов прерывания» — разные события; буферизация ведётся в модели на Python.

## 6. Апробация (тестирование)

### 6.1. Описание тестов

Тестирование состоит из двух частей:

- **Модульные тесты AST** (`tests/test_ast.py`) — проверяют, что транслятор строит корректное и человекочитаемое дерево (приоритеты операций, `while`/`if`, `interrupt function`, массивы), сверяя как структуру (`dict`), так и текстовый рендер `format_ast`.
- **Golden-тесты** (`tests/test_golden.py`, кейсы в `tests/golden/*.yml`) — прогоняют связку транслятор + модель и сверяют образ, листинг, ввод/вывод и журнал процессора с эталоном.

### 6.2. Как запускать

```
pip install -r requirements-dev.txt
pytest --update-goldens     # один раз: сгенерировать эталоны, просмотреть, закоммитить
pytest                      # обычный прогон: сверка с эталонами
```

Тесты запускаются из корня репозитория. Непрерывная интеграция настроена в `.github/workflows/ci.yml` (установка зависимостей + `pytest`).

### 6.3. Структура golden-теста

Каждый YAML описывает вход (`in_*`) и эталон (`out_*`):

- `in_source` — исходный код алгоритма;
- `in_stdin` — расписание ввода;
- `out_code` — бинарный образ (код и данные);
- `out_listing` — отладочный листинг;
- `out_output` — данные на выводе;
- `out_stdout` — итоговая сводка модели;
- `out_log` — журнал работы процессора.

Это покрывает требование «golden-тест включает алгоритм, машинный код и данные, ввод/вывод и журнал». Для тяжёлых алгоритмов полный журнал занимает сотни тысяч строк, поэтому он **адаптируется**: поле `in_log_max_lines` оставляет репрезентативный срез (голова + хвост с маркером пропуска), включающий настройку, прерывания и завершение.

## 7. Пример использования инструментальной цепочки

Трансляция и запуск программы `hello`:

```
$ python translator.py examples/hello.alg hello.bin
OK: examples/hello.alg -> hello.bin (код: 14 инструкций / 56 B, данные: 56 B, ...)

$ python machine.py hello.bin
...
OUTPUT: Hello, world!
```

Фрагмент потактового журнала (`--log`):

```
DEBUG   machine:simulation    TICK:      0 RUN  FETCH | PC=0x0000 | IR=0x0A000004 (jmp 0x0004) | R0=0 ... R7=0 ZF=0 NF=0	input []
DEBUG   machine:simulation    TICK:      1 RUN  EXEC  | PC=0x0000 | jmp 0x0004 | PC := 0x0004 | R0=0 ... R7=0 ZF=0 NF=0	input []
DEBUG   machine:simulation    TICK:      3 RUN  EXEC  | PC=0x0004 | li r7, #16384 | R7 := 16384 | ... R7=16384 ZF=0 NF=0	input []
```

Итоговая сводка:

```
--- summary ---
ticks executed: 200
instructions:   100
interrupts:     0
output (13 bytes): Hello, world!
```

Пример программы с вводом через прерывания (`sort`):

```
$ python machine.py sort.bin sort_input.txt
OUTPUT: 10 20 30      # вход: "3 30 10 20"
```

В журнале такой программы видны строки в состоянии `INT` — работа обработчика прерывания, написанного на самом языке.
